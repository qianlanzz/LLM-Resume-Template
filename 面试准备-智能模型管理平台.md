# 智能模型资源管理平台 — 面试准备文档

## 一、项目概述

面向航天领域的 AI 模型全生命周期管理平台，技术栈 Django + Vue 3，覆盖模型/算法/数据集的版本管理、自动化审核、容器化部署与语义检索，涉及 30+ 数据模型、20+ 前端页面。

---

## 二、语义向量检索引擎

### 2.1 为什么要做语义检索？

平台模型资源持续增长，原有关键词检索依赖用户精确输入模型名称或编码，召回率不足。用户往往只能描述"我需要一个做卫星故障预测的模型"，关键词匹配无法理解这种语义意图。

### 2.2 整体架构

三个核心组件：
- **RetrievalTextBuilder**：将模型元数据构建为检索文档
- **Qwen3EmbeddingProvider**：文本向量化
- **FaissIndexStore**：向量索引存储与检索

整个 VectorService 采用单例模式（`__new__` + `threading.Lock`），保证全局唯一实例。

### 2.3 文档构建细节

每个模型被构建为一段结构化文本，包含以下字段及截断限制：
- 模型名称（200字符）、编码（120）、所属组（120）
- 分类（240）、描述（800）
- 关联数据集（800）、关联算法库（800）
- 最新版本号（120）、版本状态（80）、版本说明（800）

空值字段自动剔除，关联名称通过 `dict.fromkeys()` 去重后用中文逗号拼接。

### 2.4 编码过程

- 模型：Qwen3-Embedding-0.6B，输出 1024 维向量
- 仅从本地加载（`weights/Qwen/Qwen3-Embedding-0.6B`），不走远程下载
- GPU 端使用 float16 推理，CPU 端 float32
- 推理全程包裹在 `torch.inference_mode()` 中
- 分批编码，batch_size=16
- Pooling 策略：last-token pooling（取最后一个非 padding token 的隐状态），之后 L2 归一化
- 查询编码时会注入指令前缀：`"Instruct: {instruction}\nQuery: {query}"`（非对称检索模式）
- 文档编码不加指令前缀

### 2.5 索引设计

- FAISS 索引类型：`IndexIDMap2(IndexFlatIP)`——内积相似度 + ID 映射，支持按 ID 删除
- 增量更新：upsert 采用 remove-then-add 模式，先删旧向量再加新向量
- 搜索时取 `max(top_k, candidate_pool)` 个候选，再按 similarity_threshold=0.35 过滤

**clone-on-read 无锁搜索**：
- 写入后通过 `faiss.serialize_index` / `faiss.deserialize_index` 克隆一份搜索快照 `_search_index`
- 搜索时从快照再克隆一份私有副本，整个搜索过程不持锁
- 写操作只在 `_index_store.index` 上进行，通过 RLock 保护

**原子持久化**：
1. 写入 staging 目录（命名含 pid + thread_id + uuid）
2. `os.replace()` 原子替换到最终路径
3. `os.fsync()` 刷盘目录
4. finally 块清理 staging 目录

### 2.6 自动索引更新

通过 Django Signal 实现：
- `post_save` / `post_delete` on AIModel → upsert/delete
- `post_save` / `post_delete` on ModelVersion → upsert 父模型
- `m2m_changed` on categories/datasets/algorithms → upsert（仅 post_add/post_remove/post_clear）
- 所有 handler 异常被 catch 并 log，不会阻断模型保存
- 受 `ENABLE_SEMANTIC_SEARCH` 和 `AUTO_UPDATE_EMBEDDINGS` 两个开关控制

### 2.7 可能的面试问题

**Q: 为什么选 FAISS 而不是 Milvus/Weaviate？**
A: 平台模型数量在千级别，不需要分布式向量数据库的复杂度。FAISS 单机部署、零外部依赖、内存索引性能足够，且 IndexIDMap2 支持按 ID 增删，满足增量更新需求。

**Q: 为什么用 last-token pooling 而不是 mean pooling？**
A: Qwen3-Embedding 模型官方推荐 last-token pooling，因为该模型是 decoder-only 架构，最后一个 token 的隐状态聚合了整个序列的信息，类似 CLS token 在 encoder 模型中的作用。

**Q: clone-on-read 的代价是什么？**
A: 每次搜索都要序列化/反序列化一次索引，有内存和 CPU 开销。但模型数量在千级别时索引很小（1024维 × 1000条 ≈ 4MB），这个开销可以忽略。如果数据量增长到百万级，可以改用 FAISS 的 `IndexIVFFlat` 配合读写锁替代。

**Q: 为什么不用全量重建而是增量更新？**
A: 全量重建需要加载所有模型、重新编码、重建索引，随模型数量增长耗时线性增加。增量更新只处理变更的模型，通过 Signal 自动触发，用户无感知。但也保留了 `rebuild_index()` 方法用于索引损坏时的兜底恢复。

**Q: 原子持久化为什么不直接写文件？**
A: 直接写文件如果中途崩溃（断电、OOM kill），会留下半写的损坏文件。staging + os.replace 保证要么是旧的完整文件，要么是新的完整文件，不会出现中间状态。os.fsync 确保数据真正落盘而不是停留在 OS 缓存。

**Q: similarity_threshold 0.35 是怎么定的？**
A: 内积相似度在 L2 归一化后等价于余弦相似度，范围 [-1, 1]。0.35 是经验值，在实际测试中能过滤掉大部分不相关结果，同时保留语义相近但措辞不同的模型。可以通过配置调整。

## 三、容器化部署与镜像交付

### 3.1 部署流程

基于 Docker SDK for Python 实现：
1. 清理同名旧容器（stop timeout=10s + force remove）
2. 自动检测运行环境（Docker 内 vs 宿主机），选择模型权重目录
3. 从 MinIO 下载模型权重文件到本地
4. 创建容器：注入环境变量（MODEL_NAME, SERVICE_PORT, WEIGHTS_PATH=/app/weights），挂载权重目录（只读），设置资源限制（默认 mem_limit=2g, cpu_limit=1.0 即 1e9 nano_cpus），restart_policy=unless-stopped
5. 失败时自动清理容器和临时目录

### 3.2 健康检查与自动恢复

Celery Beat 调度：
- **60s 周期**：遍历所有 status=running 的部署，检查 Docker 容器状态，如果容器不在运行则标记为 stopped/error；对健康的服务发 HTTP GET 到 health_check_url（timeout=10s）
- **10min 周期**：查找最近 1 小时内 status=failed 且 deploy_config.auto_restart=True 的部署，重置状态为 deploying 触发重新部署
- **5min 周期**：采集容器指标（CPU、内存、QPS、错误率）写入 ServiceMetrics

### 3.3 镜像导入五阶段状态机

`PENDING → IMPORTING → TAGGING → PUSHING → SUCCESS`（或任意阶段 → FAILED）

1. **Download**：如果来源是 MinIO，先下载 tar 文件到本地
2. **Importing**：`docker load -i <tar>`，正则解析输出获取镜像引用
3. **Tagging**：`docker tag <loaded_ref> <target_image>`
4. **Pushing**：如有 Harbor 凭据先 `docker login`，然后 `docker push`
5. **Register**：回填 ModelVersion.docker_image，upsert ImageAsset 记录

失败清理：删除目标镜像 tag、本地 tar 文件、MinIO 临时对象。

### 3.4 定时清理策略

- 成功任务：1 小时后清理残留文件
- 失败任务：24 小时后清理（可配置）
- 孤儿文件：48 小时后清理，交叉检查活跃任务路径避免误删
- 孤儿 MinIO 对象：扫描 `image-imports/` 前缀，同样 48 小时保留期

### 3.5 容器指标采集

- CPU：`(container_cpu_delta / system_cpu_delta) × 核数 × 100`
- 内存：`usage / limit × 100`
- QPS：最近 60 秒 inference_log 计数
- 错误率：`failed_requests / total_requests × 100`
- 平均响应时间：最近 60 秒 inference_log 的 Avg(response_time)

### 3.6 可能的面试问题

**Q: 为什么镜像导入要设计成状态机？**
A: 镜像导入是一个耗时的多步操作（下载 + load + tag + push），任何一步都可能失败。状态机让每一步的进度可追踪、可恢复，失败时能精确知道在哪一步出了问题，也方便做针对性的清理。

**Q: 为什么用 Celery 而不是直接在请求中同步执行？**
A: 镜像 load 和 push 可能耗时几分钟到几十分钟，同步执行会阻塞 HTTP 请求导致超时。Celery 异步执行，前端通过轮询任务状态获取进度。

**Q: Docker 环境检测为什么要三种方式？**
A: 不同 Docker 运行时和配置下，单一检测方式不可靠。`/.dockerenv` 是 Docker 官方标记但某些精简镜像可能没有；cgroup 信息在 cgroup v2 下格式不同；环境变量是用户可控的兜底方案。三种方式取或，提高检测准确率。

**Q: 容器资源限制 2g 内存和 1 核 CPU 够用吗？**
A: 这是默认值，实际部署时通过 deploy_config 可配置。不同模型资源需求差异很大，小模型推理 2g 足够，大模型需要用户根据实际情况调整。

**Q: 自动重启为什么只处理最近 1 小时的失败？**
A: 超过 1 小时的失败通常不是临时性问题（如网络抖动），而是配置错误或资源不足等需要人工介入的问题。自动重启只处理可能自愈的短期故障，避免无限重试浪费资源。

## 四、个性化推荐与智能分析

### 4.1 混合推荐引擎

三路融合策略：

**协同过滤（40%）**：
- 基于 Jaccard 相似度：`|交集| / |并集|`，计算用户-资源交互集合的重合度
- 最低交互门槛：用户至少 3 次交互才参与相似度计算
- 相似度阈值：0.3，低于此值不认为是相似用户
- 取 Top 5 相似用户的行为记录
- 行为权重：star=3.0, download=2.5, view=1.0
- 最终得分 = 用户相似度 × 行为权重，按资源累加

**内容推荐（30%）**：
- 从用户画像中提取资源类型偏好
- 按偏好类型分配配额（limit // 3 per type）
- 在对应类型中按热度排序推荐

**热度排序（30%）**：
- 趋势评分公式：`activity_count × 0.4 + unique_users × 0.3 + stars × 0.2 + (downloads / 100) × 0.1`
- 时间窗口：最近 7 天
- 排除用户已交互的资源

**融合去重**：按 `(resource_type, resource_id)` 去重，不足 limit 时从各路补充。

### 4.2 用户画像构建

- 资源类型偏好：统计各类型交互次数分布
- 活跃时段：统计 24 小时活动分布，取 Top 3 活跃时段
- 用户分级：new（0次）、casual（<10）、regular（<50）、power（≥50）
- 参与度（最近 7 天）：inactive（0）、low（<5）、medium（<15）、high（≥15）
- 统计窗口：最近 30 天

### 4.3 DBSCAN 异常检测

- 特征向量（5 维）：CPU、内存、磁盘、网络（发送+接收）、GPU
- 预处理：StandardScaler 标准化
- 参数：eps=0.5, min_samples=3
- 异常判定：cluster=-1 的点
- 异常评分：到数据质心的欧氏距离，归一化到 0~100
- 严重度分级：critical（>80）、high（>60）、medium（>40）、low（≤40）
- 最少需要 10 个数据点，返回 Top 10 异常

### 4.4 趋势预测

- 方法：scipy.stats.linregress 线性回归
- 最少 7 个数据点
- 预测未来 7 天，加入高斯噪声（std × 0.1）模拟真实波动
- 预测值裁剪到 [0, 100]
- 输出：斜率、R²、趋势方向
- 洞察生成：7 日滚动均值对比，变化幅度 >30% 才报告

### 4.5 系统监控

- 采集：psutil（CPU 1s 间隔、内存 MB、磁盘 GB、网络 MB、负载、进程数），可选 GPUtil
- 告警阈值：CPU warning>90% critical>95%，内存 warning>85% critical>95%，磁盘 warning>85% critical>95%
- 告警去重：同标题未解决告警 1 小时内不重复创建
- 服务健康检查：数据库（SELECT 1）、Redis（set/get/delete 循环）、Celery（inspect.stats 查 worker 数）、MinIO（bucket 存在性 + 对象列表）

### 4.6 可能的面试问题

**Q: 为什么用 Jaccard 而不是余弦相似度做协同过滤？**
A: 用户-资源交互是二值关系（交互过/没交互过），不涉及评分。Jaccard 专门衡量集合重合度，语义清晰且计算简单。余弦相似度更适合有评分或权重的场景。

**Q: 三路推荐的权重 40/30/30 怎么定的？**
A: 协同过滤权重最高是因为"相似用户的行为"在模型推荐场景下最有参考价值——做类似任务的人大概率需要类似的模型。内容推荐和热度作为补充，防止冷启动和信息茧房。权重可以通过 A/B 测试进一步调优。

**Q: 冷启动问题怎么解决？**
A: 新用户交互不足 3 次时协同过滤不生效，此时依赖内容推荐（基于用户注册时的角色/部门推断偏好）和热度排序兜底。随着用户交互积累，协同过滤逐渐发挥作用。

**Q: DBSCAN 的 eps=0.5 和 min_samples=3 怎么选的？**
A: 数据经过 StandardScaler 标准化后，eps=0.5 大约对应 0.5 个标准差的距离，能捕捉到明显偏离正常范围的点。min_samples=3 是因为系统指标采集频率不高（5 分钟一次），太大会导致正常波动也被标记为异常。实际部署中可以根据误报率调整。

**Q: 线性回归做趋势预测准确吗？**
A: 线性回归只能捕捉线性趋势，对周期性波动（如白天高晚上低）效果不好。但在容量规划场景下，我们关心的是长期增长趋势而非短期波动，线性回归足够。R² 值可以衡量拟合质量，R² 太低时可以提示"趋势不明显"。

**Q: 为什么异常检测用 DBSCAN 而不是 Isolation Forest？**
A: DBSCAN 不需要预设异常比例，适合不知道异常占比的场景。Isolation Forest 需要指定 contamination 参数。另外 DBSCAN 能发现任意形状的簇，对系统指标这种多维数据更灵活。

## 五、其他可能被追问的点

### 5.1 模型版本状态机

7 个状态：train_done → test_pass → online → offline，以及 test_fail、rollback_test、rollback_train。

状态变更时：
- 自动记录 ModelVersionStatusHistory（from_status, to_status, operator, reason）
- test_pass 触发 WebSocket 通知（Django Channels），推送给模型创建者 + 版本创建者 + 审核角色用户
- 自动清理过期的 reason 字段

### 5.2 发布审核

8 项检查：
- Required（阻断发布）：版本状态合法性、模型描述 Markdown 结构（必须包含"模型介绍/模型性能/模型使用"三个标题）、至少一个分类、Docker 镜像地址
- Warning（不阻断）：关联数据集、MinIO 文档/测试报告、镜像可达性、关联算法

### 5.3 数据批量迁移

- 导出：ZIP 包含 manifest.json + payload.json + MinIO 二进制制品，对象通过 (bucket, object_name) 去重
- 导入：支持预览 → 选择性导入，层级分类通过迭代式拓扑排序解析父子依赖（while 循环，每轮处理已解析父节点的子分类，无进展时跳出防死循环），全程 transaction.atomic

### 5.4 技术选型理由

| 组件 | 选型 | 理由 |
|------|------|------|
| 向量索引 | FAISS | 千级数据量，单机足够，零外部依赖 |
| Embedding | Qwen3-Embedding-0.6B | 中文语义理解好，0.6B 参数量轻量可本地部署 |
| 任务队列 | Celery + Redis | Django 生态成熟，支持定时任务和异步执行 |
| 对象存储 | MinIO | S3 兼容，私有化部署，支持预签名上传和分片上传 |
| 容器管理 | Docker SDK | Python 原生集成，比 subprocess 调 CLI 更可靠 |
| 实时通知 | Django Channels | 与 Django 深度集成，WebSocket 支持 |
| 前端 | Vue 3 + Element Plus | 组件丰富，TypeScript 支持好 |
